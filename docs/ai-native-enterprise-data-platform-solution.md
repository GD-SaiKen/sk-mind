# AI 原生企业数据平台产品方案

版本：草案 v0.6

日期：2026-06-13

## 文档说明

本文档用于描述一个面向中小型制造企业的 AI 原生数据平台方案。平台目标是一方面汇总企业内部来自业务系统、数据库、文件、Excel、共享盘和接口的数据，另一方面通过轻量语义本体、数据治理、指标口径和 Agent 行动控制，让 Agent 能够基于可信企业数据提供可解释、可追踪、可控的决策信息。

本文档当前处于产品方案设计阶段，重点描述平台定位、目标客户、总体架构、核心能力、语义本体模型、Agent 服务边界和能力成熟度，不进入具体工程实现计划。

## 1. 方案总览

### 1.1 平台定位

本平台定位为面向企业客户的 AI 原生数据平台。

平台一方面汇总企业内部来自业务系统、数据库、文件、Excel、共享盘和接口的数据，形成统一、可追踪、可治理的数据底座；另一方面通过轻量语义本体、指标口径、规则模型和行动控制，为 Agent 提供可理解、可查询、可解释、可控使用的数据环境。

平台目标不是简单建设一个数据仓库，也不是单独建设一个智能问答应用，而是建设一套让 Agent 能可靠使用企业数据的基础设施。

更准确地说，本平台是一个连接“企业数据资产”和“AI Agent 决策服务”的中间层。它将原本分散、异构、质量不一、语义不清的数据，转化为 Agent 可以理解和使用的企业语义资产。

### 1.2 目标客户

第一批目标客户为中小型制造企业。

此类企业通常具备以下特点：

- 已经建设部分业务系统，例如 ERP、MES、采购、考勤、财务、仓储或生产相关系统。
- 各系统建设时间、供应商、数据结构和开放能力不一致。
- 除业务系统外，仍存在大量 Excel、线下文件、共享盘和人工维护数据。
- 数据分散在部门、岗位、系统和历史文件中，缺少统一目录、质量治理和业务语义。
- 经营分析、异常判断和管理决策仍依赖人工汇总、人工解释和经验判断。

对于此类客户，如果 Agent 直接访问原始表、Excel 文件或接口数据，容易因为字段含义不清、口径不一致、数据质量不稳定而给出不可靠结论。因此，平台首先需要建设可信数据底座和语义控制面，再让 Agent 使用这些数据。

### 1.3 核心价值

平台的核心价值可以概括为四个方面。

第一，数据汇总。

平台接入 ERP、MES、采购、考勤、Excel、共享盘、数据库和 API 等多类数据源，形成统一企业数据资产。对于历史 Excel 文件和共享盘数据，平台支持以批量方式接入，不要求用户逐个文件手工上传。

第二，数据治理。

平台保留数据来源、同步批次、血缘、质量状态和权限信息，让数据可追踪、可审计、可治理。数据不会在进入平台后立即被强行统一为复杂业务模型，而是先进入 Raw 层保留原貌，再通过 Clean 层进行校验和轻标准化。

第三，语义统一。

平台通过对象、属性、关系、事件状态、规则、指标、行动和数据映射，将数据从“表字段”转化为可理解的业务语义。Agent 不直接面对零散数据表，而是通过语义层理解企业业务对象、指标口径、规则边界和数据来源。

第四，Agent 服务。

平台让 Agent 基于可信数据进行查询、分析、解释、建议和预警，并通过行动分级控制风险。初期 Agent 主要提供只读查询、分析解释和建议，不默认自动写回 ERP、MES、采购、考勤等核心业务系统。

### 1.4 总体架构

平台总体架构从数据生命周期和 Agent 使用链路出发，可以分为七层：

```text
数据源
→ 数据接入层
→ Raw 原始层
→ Clean 清洗层
→ Business / Analytics 层
→ Semantic 语义层
→ Agent 服务层
```

各层职责如下。

| 层级 | 主要职责 |
| --- | --- |
| 数据源 | 包括 ERP、MES、采购、考勤、Excel、共享盘、API、数据库等企业内部数据来源 |
| 数据接入层 | 管理连接器、采集任务、同步批次、接入日志、失败重试和来源系统元数据 |
| Raw 原始层 | 保留原始数据、来源追踪、同步批次和审计信息，支持重放和追溯 |
| Clean 清洗层 | 对数据进行校验、去重、轻标准化、质量标记和基础治理 |
| Business / Analytics 层 | 建立统一业务对象、事实表、维度表、指标视图和分析服务 |
| Semantic 语义层 | 管理轻量语义本体、数据映射、规则、指标、权限和 Agent 行动边界 |
| Agent 服务层 | 提供自然语言问答、查询规划、工具调用、结果解释、建议生成、人工确认和审计 |

这一架构借鉴数据湖仓分层思想，但并不把分层本身作为目的。分层的核心目的，是让数据从“接得进来”逐步变成“查得到、信得过、讲得清、用得稳”。

### 1.5 轻量语义本体

平台采用轻量语义本体思想，而不是第一阶段建设重型知识图谱平台。

语义本体层包括以下子模型：

| 子模型 | 作用 |
| --- | --- |
| 对象模型 | 定义企业业务世界里有哪些对象，例如客户、供应商、物料、产品、订单、工单、设备、员工等 |
| 属性模型 | 定义对象有哪些标准属性，例如编码、名称、状态、时间、数量、金额、单位等 |
| 关系模型 | 定义对象之间如何连接，例如组成、归属、供应、消耗、负责、计算自、映射到等 |
| 业务关系图谱模型 | 定义具体业务对象实例之间的关系边，例如某订单包含哪些订单行、某产品使用哪些物料、某工单消耗哪些库存 |
| 事件状态模型 | 定义对象什么时候发生变化、状态如何流转、哪些事件会触发规则或行动 |
| 规则模型 | 定义校验、判断、预警、解释、权限和行动约束 |
| 指标模型 | 定义经营指标、计算公式、分析维度、口径规则和可信等级 |
| 数据映射模型 | 定义语义对象、属性、关系、指标如何映射到来源系统、表、字段、文件和 API |
| 行动模型 | 定义 Agent 能做什么、不能做什么、什么动作需要确认或审批 |
| 对象行动模型 | 定义每类业务对象可被执行哪些动作、动作风险等级、权限要求、确认要求和审计要求 |

轻量语义本体层的核心目的，是让 Agent 知道：

- 企业里有哪些业务对象。
- 这些对象有哪些属性。
- 对象类型之间如何关联。
- 具体对象实例之间形成了哪些业务关系路径。
- 每类对象允许执行哪些查询、分析、解释、建议、协同或写入动作。
- 指标如何计算。
- 规则如何判断。
- 真实数据在哪里。
- 哪些数据可以使用。
- 哪些行动可以执行。
- 哪些行动必须经过人工确认。

因此，本平台借鉴本体建模中的对象、属性、关系和规则思想，但不要求第一阶段采用完整 RDF、OWL、三元组存储或复杂推理机。

### 1.6 Agent 使用边界

Agent 使用企业数据时必须分级治理。平台将 Agent 行动分为五个等级：

| 等级 | 行动类型 | 示例 |
| --- | --- | --- |
| L0 只读 | 查询、检索、查看 | 查询库存、查看订单、检索客户信息 |
| L1 分析 | 计算、对比、解释 | 计算指标、对比趋势、解释异常 |
| L2 建议 | 生成建议、预警、待办草稿 | 生成补货建议、风险提醒、待办草稿 |
| L3 协同 | 发起确认、提交审批、创建任务 | 创建治理工单、提交审批、请求业务确认 |
| L4 写入 | 修改数据、回写系统、触发业务动作 | 修改 ERP 数据、关闭工单、回写采购状态 |

初始边界建议如下：

- 默认开放 L0-L2。
- L3 必须由用户确认。
- L4 不默认自动开放。
- 涉及 ERP、MES、采购、考勤等核心业务系统写入时，必须经过审批和审计。

这一边界设计的目的，是让 Agent 先成为可信的数据查询、分析和解释助手，再逐步参与协同和决策闭环，而不是一开始就自动执行高风险业务动作。

### 1.7 能力成熟度

平台能力不应被理解为一次性交付的功能清单，而应被理解为逐步成熟的能力体系。

#### 能力成熟度 1：可信数据底座

目标是让数据可接入、可追踪、可治理、可查询。

核心能力包括数据源接入、Raw/Clean 分层、来源追踪、质量标记、基础数据目录和权限控制。

在这一阶段，Agent 主要做只读查询和基础解释。

#### 能力成熟度 2：语义治理与指标分析

目标是让数据能被业务一致理解。

核心能力包括对象模型、属性模型、关系模型、规则模型、指标模型、数据映射、语义目录和指标口径管理。

在这一阶段，Agent 可以做分析、对比、归因和可解释建议。

#### 能力成熟度 3：Agent 决策辅助闭环

目标是让 Agent 安全参与业务决策。

核心能力包括行动模型、工具编排、人工确认、预警建议、协同流程、审计反馈和持续优化。

在这一阶段，Agent 可以触发协同行动，但高风险写入仍需要审批。

### 1.8 方案边界

为保持方案可落地，平台在初期需要明确边界。

第一，不先做万能 Agent。

平台优先建设 Agent 可用的数据底座和语义控制面，而不是直接让 Agent 面对所有原始数据和系统接口。

第二，不先做重型本体。

平台采用轻量语义元模型，先建设可维护的对象、属性、关系、规则、指标和映射体系。知识图谱、复杂推理和完整本体工程可以作为后续能力增强。

第三，不替代原业务系统。

平台负责汇总、治理、建模和服务数据，不直接替代 ERP、MES、采购、考勤、财务等业务系统。

第四，不默认自动写回。

Agent 可以查询、分析、解释和建议，但涉及核心业务系统写入、审批、关闭、修改等高风险动作时，必须经过人工确认、权限校验和审计。

## 2. 客户现状与问题

### 2.1 数据源分散

中小型制造企业的数据通常分散在多个系统和文件环境中。典型数据源包括 ERP、MES、采购系统、考勤系统、财务系统、仓储系统、生产设备数据、Excel 文件、共享盘文件以及部门自建台账。

这些数据源往往具有以下特点：

- 系统建设阶段不同，数据结构和接口能力不一致。
- 同一类业务对象在不同系统中存在不同名称和编码。
- 历史 Excel 文件数量多、模板多、分布散，无法依赖人工逐个上传。
- 部门之间形成各自的数据口径和维护方式。
- 部分关键数据只存在于文件、人工台账或共享盘中，未进入正式业务系统。

数据源分散带来的直接后果是，企业虽然拥有大量数据，但数据无法被统一发现、统一管理和统一使用。对于 Agent 而言，问题更明显：Agent 不知道哪些数据存在、哪些数据可信、哪些字段代表真实业务含义，也不知道不同系统之间的数据如何关联。

### 2.2 数据质量不稳

企业内部数据长期由多个系统、部门和人员共同维护，数据质量不稳定是普遍现象。

典型问题包括：

| 问题类型 | 表现 | 影响 |
| --- | --- | --- |
| 字段不统一 | 同一含义字段在不同系统中名称不同 | 难以建立统一查询和指标口径 |
| 编码不一致 | 物料、客户、供应商等主数据编码规则不同 | 跨系统关联困难 |
| 数据缺失 | 关键字段为空，历史文件信息不完整 | 指标计算和风险判断不可靠 |
| 重复数据 | 同一对象在多个系统或文件中重复维护 | 统计结果失真 |
| 时间不一致 | 数据更新时间、业务发生时间和录入时间混用 | 趋势分析和状态判断容易出错 |
| 口径冲突 | 不同部门对同一指标有不同计算方式 | 管理层难以形成统一判断 |
| 来源不清 | 数据来自哪里、何时同步、是否被加工不明确 | 无法审计和追溯 |

因此，平台不能假设接入后的数据天然可用。需要在 Raw 层保留原貌和来源，在 Clean 层进行校验和轻标准化，在 Business / Analytics 层逐步形成可服务的数据资产。

### 2.3 业务语义缺失

传统数据平台通常关注表、字段、任务和报表，但 Agent 使用数据时更需要业务语义。

例如：

- “料号”“物料编码”“存货编码”“SKU”是否指向同一类业务对象。
- “订单”“销售订单”“客户订单”在不同系统中是否一致。
- “库存”指现存量、可用量、账面量，还是可承诺量。
- “交付率”“良率”“库存周转”等指标的计算口径是什么。
- 某个字段可以被 Agent 查询、解释、推理，还是需要权限限制。

如果缺少语义层，Agent 只能根据字段名、表名和自然语言猜测业务含义，容易出现误查、误算和误解释。对于企业决策场景，这种不确定性不可接受。

因此，本平台需要在数据层之上建设轻量语义本体层，用对象、属性、关系、事件状态、规则、指标、数据映射和行动模型来表达企业业务语义。

### 2.4 决策依赖人工

在很多中小型制造企业中，管理分析和业务决策仍高度依赖人工。

常见工作方式包括：

- 从多个系统导出数据，再由人员在 Excel 中汇总。
- 由业务负责人根据经验解释异常原因。
- 报表口径依赖个人理解，缺少统一规则。
- 发现数据问题后，通过线下沟通确认真实情况。
- 管理层提出问题后，需要等待数据人员或业务人员临时分析。

这种方式的问题在于效率低、口径不稳、过程不可追溯，也难以支撑 Agent 自动化分析。

平台的目标不是让 Agent 替代业务人员，而是让 Agent 在可信数据和语义规则约束下，帮助业务人员更快完成查询、分析、解释、预警和建议生成。

## 3. 总体架构设计

### 3.1 架构总览

平台总体架构采用“数据分层 + 语义控制面 + Agent 服务层”的组合方式。

```text
企业数据源
→ 数据接入层
→ Raw 原始层
→ Clean 清洗层
→ Business 业务模型层
→ Analytics 分析服务层
→ Semantic 语义本体层
→ Agent 服务层
```

其中，数据分层负责数据生命周期管理，语义控制面负责业务含义和治理边界，Agent 服务层负责将数据转化为可解释的决策信息。

可以将平台理解为三条能力线：

| 能力线 | 作用 |
| --- | --- |
| 数据生产线 | 从数据源接入数据，经过 Raw、Clean、Business、Analytics 层形成可信数据资产 |
| 语义治理线 | 定义对象、属性、关系、规则、指标、行动和数据映射，形成业务语义地图 |
| Agent 消费线 | 基于语义层定位数据、执行查询分析、生成解释建议，并受权限和行动边界约束 |

### 3.2 数据接入层

数据接入层负责连接企业内部多类数据源，并将数据安全、可追踪地接入平台。

主要支持的数据源类型包括：

| 数据源类型 | 示例 | 接入方式 |
| --- | --- | --- |
| 文件类 | Excel、CSV、共享盘文件、本地文件夹 | 文件扫描、模板识别、批量导入 |
| 数据库类 | ERP 数据库、MES 数据库、业务系统数据库 | 只读连接、定时同步、增量采集 |
| API 类 | 采购系统、考勤系统、第三方 SaaS | API 拉取、分页同步、接口鉴权 |
| CDC 类 | 支持变更日志的数据库 | 变更捕获、增量同步 |
| 手工类 | 管理台录入、临时导入 | 受控录入、导入校验、审计记录 |

接入层的核心对象包括：

- 来源系统：描述数据来自哪个业务系统、文件环境或接口。
- 连接器：描述如何连接来源系统。
- 来源对象：描述来源系统中的表、文件、接口或资源。
- 同步任务：描述何时、如何、以什么策略同步数据。
- 同步批次：记录每次同步结果、数据量、状态、错误和时间。
- Raw Dataset：接入后形成的原始数据集。

对于 Excel 和共享盘数据，平台不要求用户逐个上传历史文件。更合理的方式是：每类 Excel 模板先选取少量样本进行人工校验和映射确认，确认无误后由平台批量扫描和处理同类文件。

### 3.3 数据分层

平台采用分层数据架构，建议分为 Raw、Clean、Business、Analytics 四层。

| 层级 | 目标 | 主要内容 |
| --- | --- | --- |
| Raw 层 | 保留原始数据和来源 | 来源系统原始表、文件解析结果、同步批次、原始字段、来源追踪 |
| Clean 层 | 提升数据可用性 | 类型校验、格式标准化、去重、质量标记、轻标准化字段 |
| Business 层 | 形成统一业务对象 | 客户、供应商、物料、产品、订单、工单、库存、设备、员工等业务对象 |
| Analytics 层 | 支撑分析和指标服务 | 事实表、维度表、指标视图、分析宽表、查询 API |

Raw 层不追求业务统一，重点是保真、可追踪、可重放。Clean 层不做过度业务建模，重点是校验和轻标准化。Business 层开始建立统一业务对象和关系。Analytics 层面向指标、分析和 Agent 查询提供服务。

这种分层方式可以降低接入初期的不确定性，避免一开始就要求所有系统和文件完全符合统一业务模型。

### 3.4 语义与 Agent 层

语义与 Agent 层是本平台区别于传统数据平台的关键部分。

语义层负责描述：

- 企业有哪些业务对象。
- 对象有哪些标准属性。
- 对象之间有哪些关系。
- 对象有哪些状态和事件。
- 指标如何计算。
- 规则如何判断。
- 语义如何映射到真实表、字段、文件和 API。
- Agent 能执行哪些行动，哪些行动需要确认或审批。

Agent 层负责基于语义层使用数据：

```text
用户问题
→ 语义解析
→ 识别对象、指标、时间、范围
→ 展开关系路径和规则口径
→ 通过数据映射定位真实数据
→ 检查权限、质量、血缘和时效
→ 执行查询、计算、对比或解释
→ 输出结果、依据、不确定性和建议
```

语义层不替代数据仓库，也不要求所有数据变成知识图谱。它是一个轻量语义控制面，负责让数据对 Agent 可理解、可追踪、可解释、可控。

### 3.5 治理与安全

治理与安全贯穿平台全链路。

核心治理能力包括：

| 治理能力 | 内容 |
| --- | --- |
| 元数据管理 | 数据源、数据集、字段、任务、指标、规则、语义对象 |
| 数据血缘 | 数据从来源到 Raw、Clean、Business、Analytics、Semantic 的流转关系 |
| 数据质量 | 完整性、唯一性、准确性、一致性、及时性、冲突状态 |
| 权限控制 | 租户、角色、数据集权限、字段权限、敏感级别、脱敏策略 |
| 审计记录 | 用户查询、Agent 调用、工具执行、数据访问、人工确认和审批 |
| 可信等级 | 对数据、规则、指标、映射和 Agent 结果标记可信程度 |

对于 Agent，治理与安全尤其重要。平台必须保证 Agent 的回答不是“凭空生成”，而是能够说明使用了哪些数据、哪些指标口径、哪些规则、哪些时间范围，以及哪些结果存在不确定性。

## 4. 核心模块设计

### 4.1 数据源与连接器

数据源与连接器模块负责统一管理企业内部数据来源和连接方式。

主要能力包括：

- 来源系统登记：记录系统名称、类型、负责人、业务领域、访问方式和数据敏感级别。
- 连接器管理：支持文件、数据库、API、CDC、手工导入等连接类型。
- 连接权限管理：记录账号、密钥、访问范围、只读策略和过期时间。
- 连接状态检测：检查连接是否可用、接口是否异常、共享盘是否可访问。
- 来源对象管理：登记来源系统中的表、接口、文件目录、Excel 模板和字段结构。

该模块的设计原则是：优先只读接入业务系统，不直接修改源系统数据；对于高风险系统，接入权限应最小化，并保留访问审计。

### 4.2 数据接入与同步

数据接入与同步模块负责将数据从来源系统接入平台。

主要能力包括：

- 同步任务配置：配置同步对象、频率、增量策略、字段范围和目标数据集。
- 同步批次管理：记录每次同步的开始时间、结束时间、数据量、状态和错误信息。
- 失败重试与告警：对连接失败、字段变化、权限失效、文件缺失等情况进行记录和提醒。
- Schema 变化检测：识别来源字段新增、删除、类型变化和结构漂移。
- 批量文件处理：支持本地文件夹和企业共享盘中的历史 Excel 文件批量处理。
- 接入日志审计：保留接入过程中的任务、用户、时间、文件、记录数和错误明细。

对于中小制造企业，Excel 和共享盘数据是重要接入对象。平台应支持“样本校验 + 规则确认 + 批量处理”的接入方式，以降低历史数据迁移成本。

### 4.3 数据处理与质量

数据处理与质量模块负责将原始数据转化为更可用的数据资产。

处理范围包括：

- 字段类型识别和转换。
- 日期、数量、金额、单位、编码、文本格式标准化。
- 空值、重复值、异常值、非法枚举检测。
- 主键和候选键识别。
- 跨系统编码冲突标记。
- 数据质量评分和质量报告。
- 清洗结果、错误记录和问题样本导出。

本平台在初期不建议做过重的数据清洗。更合理的策略是“校验 + 轻标准化 + 质量标记”。也就是说，对于无法确认的冲突数据，不强行修正，而是保留差异、标记质量状态，并在语义层或治理流程中处理。

### 4.4 数据目录与血缘

数据目录与血缘模块负责让数据资产可发现、可理解、可追踪。

主要能力包括：

- 数据资产目录：展示数据源、数据集、表、字段、文件、API 和指标。
- 字段说明：记录字段名称、类型、业务含义、来源、质量状态和敏感等级。
- 数据血缘：记录数据从来源到 Raw、Clean、Business、Analytics、Semantic 的加工链路。
- 影响分析：当来源字段、规则或指标变化时，识别受影响的数据集、指标和 Agent 能力。
- 使用统计：记录数据集被哪些报表、指标、Agent 或用户使用。
- 可信状态：显示数据是否可用、是否过期、是否存在质量问题。

数据目录不是简单的表清单，而是连接数据资产、语义模型和 Agent 使用过程的桥梁。

### 4.5 语义建模工作台

语义建模工作台用于维护轻量语义本体。

主要维护对象包括：

- 业务对象：客户、供应商、物料、产品、订单、工单、库存、设备、员工等。
- 对象属性：编码、名称、状态、时间、数量、金额、组织、版本等。
- 语义关系：组成、归属、供应、消耗、负责、计算自、映射到等。
- 事件状态：创建、审核、变更、冻结、完工、关闭等事件和状态流转。
- 规则：质量规则、业务判断规则、权限规则、解释规则和行动规则。
- 指标：指标定义、公式、维度、时间粒度、口径规则和可信等级。
- 数据映射：语义对象、属性、关系、指标到来源系统、表、字段、文件和 API 的映射。
- 行动边界：Agent 可执行动作、风险等级、确认要求和审计要求。

该工作台面向业务专家、数据人员和实施人员，不应设计成复杂本体工程工具。它的目标是让企业能够逐步维护业务语义，而不是要求业务人员理解 RDF、OWL 或三元组。

### 4.6 指标与分析服务

指标与分析服务负责统一经营指标口径，并为用户和 Agent 提供分析能力。

主要能力包括：

- 指标定义：定义指标名称、业务含义、公式、分子分母、适用范围。
- 维度管理：定义时间、组织、客户、供应商、物料、产品、设备、仓库等分析维度。
- 指标血缘：记录指标依赖哪些对象、属性、关系、规则和数据集。
- 口径管理：记录过滤条件、排除条件、时间粒度、缺失值处理和异常值处理方式。
- 查询服务：为看板、报表、自然语言问答和 Agent 提供指标查询接口。
- 解释模板：让 Agent 能说明指标如何计算、来自哪里、是否可信。

该模块的重点不是建设完整 BI 工厂，而是先统一高价值、高复用、高争议的指标口径。

### 4.7 Agent 服务层

Agent 服务层负责将平台数据和语义能力转化为用户可用的决策信息服务。

核心能力包括：

- 自然语言理解：识别用户问题中的对象、指标、时间、范围和意图。
- 查询规划：基于语义关系、指标口径和数据映射生成查询路径。
- 工具调用：调用数据查询、指标计算、规则判断、报告生成等工具。
- 可信检查：在执行前检查权限、质量、血缘、时效和行动等级。
- 结果解释：输出结果时说明数据来源、指标口径、规则依据和不确定性。
- 建议生成：基于规则和指标生成可解释建议，而不是直接替用户决策。
- 人工确认：高风险动作进入确认、审批和审计流程。

Agent 服务层的关键原则是：Agent 不绕过语义层直接访问数据，不绕过权限直接调用工具，不绕过人工确认执行高风险动作。

### 4.8 管理控制台

管理控制台面向平台管理员、实施人员和数据治理人员。

主要能力包括：

- 数据源管理。
- 接入任务管理。
- 数据集和字段管理。
- 数据质量报告。
- 语义模型管理。
- 权限和角色管理。
- Agent 能力和工具权限管理。
- 运行监控和告警。
- 审计日志查询。
- 治理工单和问题跟踪。

管理控制台的设计目标是让平台可运营、可治理、可审计，而不是只提供一次性数据导入能力。

## 5. 轻量语义本体设计

### 5.1 元模型总览

轻量语义本体层是平台的语义控制面，核心元模型包括十类对象：

| 元模型对象 | 作用 |
| --- | --- |
| BusinessObject | 业务对象定义 |
| ObjectProperty | 对象属性定义 |
| SemanticRelation | 对象类型之间的语义关系定义 |
| BusinessGraphEdge | 对象实例之间的业务关系图谱边 |
| BusinessEvent | 事件与状态变化定义 |
| BusinessRule | 校验、判断、约束和解释规则 |
| Metric | 经营指标与计算口径 |
| DataMapping | 语义到数据资产的映射 |
| AgentAction | Agent 能力和执行边界 |
| ObjectActionPolicy | 对象上的行动权限、风险等级和确认要求 |

它们之间的关系可以概括为：

```text
对象是中心
属性描述对象
关系定义对象类型如何连接
图谱边记录对象实例之间的实际关系
事件改变对象状态
规则提供判断和约束
指标提供经营口径
数据映射连接真实数据
行动模型控制 Agent 有哪些能力
对象行动策略控制什么对象上能执行什么行动
```

其中，`SemanticRelation` 与 `BusinessGraphEdge` 是一组配套模型。前者定义“客户可以下达销售订单、销售订单可以包含订单行、BOM 可以使用物料”这类关系类型；后者记录“客户 A 下达了订单 SO-001、订单 SO-001 包含订单行 1、产品 P-001 使用物料 M-001”这类具体关系实例。Agent 查询和推理时，先理解关系类型，再沿着业务关系图谱边找到真实业务对象。

同样，`AgentAction` 与 `ObjectActionPolicy` 也是一组配套模型。前者定义 Agent 具备查询、分析、解释、建议、协同、写入等能力；后者定义这些能力能否作用在某类对象上，以及是否需要权限、规则校验、人工确认和审计。

### 5.2 对象与属性模型

对象模型定义企业业务世界中有哪些对象。平台建议采用对象分类体系，而不是直接把所有对象列成平铺清单。

对象分类包括：

| 分类 | 含义 | 示例 |
| --- | --- | --- |
| 主数据对象 | 相对稳定、被多个过程复用 | 客户、供应商、物料、产品、员工、设备 |
| 交易对象 | 记录业务往来和单据 | 销售订单、采购订单、生产订单、入库单 |
| 资源对象 | 可被占用、消耗或调度 | 人员、设备、产线、物料、库存、时间 |
| 过程对象 | 表达业务活动和任务 | 采购过程、生产过程、质检过程、维修过程 |
| 事件状态对象 | 表达发生了什么和状态如何变化 | 创建、审核、开工、完工、关闭 |
| 指标对象 | 表达经营口径和计算结果 | 交付率、良率、库存周转、产能利用率 |
| 规则对象 | 表达约束、判断和预警条件 | 安全库存规则、质检合格规则 |
| 数据对象 | 表达真实数据资产和来源 | 来源系统、数据表、字段、文件、接口 |

对象定义模板应包括：

- 对象编码、名称、分类、业务定义和适用范围。
- 别名、同义词、上位对象、下位对象和业务边界。
- 主标识、候选标识、组织范围、版本规则和唯一性规则。
- 生命周期状态和状态流转约束。
- 必备属性、可选属性、扩展属性和敏感属性。
- 可作为哪些关系的主体或客体。
- 来源系统、来源对象、主键映射和字段映射。
- 数据负责人、可信等级、权限等级和 Agent 可用性。

属性模型定义对象有哪些信息。属性分类包括标识属性、描述属性、状态属性、时间属性和度量属性。

属性定义模板应包括：

- 属性编码、名称、所属对象、分类和业务定义。
- 别名、同义词、单位含义和业务边界。
- 数据类型、枚举字典、格式规则和取值范围。
- 是否必填、唯一性、质量要求和冲突处理。
- 来源系统、来源字段、转换规则和字段可信度。
- 是否可查询、可解释、可用于推理，以及权限和敏感等级。

### 5.3 关系与事件模型

关系模型定义对象之间如何连接。

关系类型字典包括：

| 关系类型 | 作用 | 示例 |
| --- | --- | --- |
| 结构关系 | 描述静态结构 | 组成、包含、归属、上下级、版本、替代 |
| 交易关系 | 描述业务往来 | 购买、销售、供应、结算、收发、转移 |
| 资源关系 | 描述资源使用 | 占用、消耗、分配、释放、调度、产出 |
| 过程关系 | 描述流程和状态变化 | 前置、后置、触发、依赖、流转、阻塞 |
| 责任关系 | 描述组织和人员责任 | 负责、审批、执行、归口、拥有、管理 |
| 规则关系 | 描述规则如何作用于对象 | 约束、校验、触发、命中、适用、例外 |
| 指标关系 | 描述指标如何形成 | 计算自、聚合自、影响、分解、归因 |
| 数据关系 | 描述语义如何落到真实数据 | 来源于、映射到、派生自、同步自、关联字段 |

关系定义模板应包括：

- 关系编码、名称、分类、业务定义、别名。
- 主体对象类型、客体对象类型、方向、反向关系和基数约束。
- 是否层级、是否可传递、是否对称、是否可推导、是否随时间变化。
- 来源系统、来源字段、关联键、同步方式和可信等级。
- 是否对 Agent 可见，是否可用于查询、推理和解释。

事件状态模型描述业务对象何时发生变化、状态如何流转。

事件定义模板应包括：

- 事件编码、名称、分类、适用对象和业务定义。
- 发生时间、触发主体、触发来源、关联对象和业务原因。
- 前置状态、目标状态、允许条件、禁止条件和回退规则。
- 影响对象、属性、关系、指标和数据集。
- 是否触发校验、规则、指标刷新、预警或 Agent 建议。
- 来源系统、可信等级、权限等级和审计要求。

事件状态模型不等于流程引擎。它先负责描述事实和状态，后续可以与审批流、工单系统或工作流系统集成。

### 5.4 规则与指标模型

规则模型定义什么时候算正常、什么时候算异常、什么时候需要判断、什么时候允许 Agent 行动。

规则分类包括：

- 结构规则：必填、唯一、类型、枚举、关系基数。
- 质量规则：空值、重复、异常值、跨系统冲突、过期数据。
- 业务判断规则：状态判断、风险判断、优先级判断、建议等级。
- 指标规则：公式、过滤条件、时间范围、排除条件。
- 行动规则：Agent 是否可查询、提醒、建议、协同或写入。
- 权限规则：谁可以看、问、导出、解释或触发行动。
- 冲突规则：多来源、多口径、多版本不一致时如何处理。
- 解释规则：Agent 回答时如何说明依据和不确定性。

指标模型定义经营口径和计算逻辑。

指标模型包括：

- 基础度量：数量、金额、工时、次数、时长、重量。
- 业务指标：交付率、良率、库存周转、产能利用率等。
- 分析维度：时间、组织、客户、供应商、物料、产品、设备。
- 计算口径：公式、过滤条件、统计范围、时间粒度。
- 解释依据：数据来源、规则版本、质量状态、可信度。

指标定义模板应包括指标编码、名称、分类、业务定义、公式、维度、时间粒度、依赖对象、依赖字段、口径规则、负责人、版本、可信等级和 Agent 可用性。

### 5.5 数据映射模型

数据映射模型连接语义层和真实数据资产。

核心映射类型包括：

| 映射类型 | 作用 |
| --- | --- |
| 对象映射 | 语义对象对应哪些来源系统对象，例如 ERP 表、MES 表、Excel 模板、API 资源 |
| 属性映射 | 标准属性对应哪些来源字段，包括字段名、类型、单位、枚举和转换规则 |
| 关系映射 | 语义关系如何从数据中获得，例如外键、关联键、中间表、规则推导或人工确认 |
| 指标映射 | 指标口径依赖哪些对象、字段、过滤条件、聚合逻辑和时间粒度 |

映射定义应记录：

- 语义端：对象、属性、关系、指标和业务定义版本。
- 数据端：来源系统、来源表、文件、接口、字段、主键。
- 转换端：类型转换、单位换算、枚举映射、清洗规则、关联规则。
- 治理端：负责人、可信等级、质量要求、生效状态、变更记录。
- 血缘端：上游数据集、下游数据集、加工作业、同步批次。
- 权限端：数据等级、敏感字段、访问角色、脱敏方式。
- Agent 端：是否可查询、可推理、可解释、查询优先级和不确定时处理方式。

初期不要求把所有字段都语义化。优先映射对象主标识、关键属性、核心关系、指标依赖字段和 Agent 可查询入口。

### 5.6 行动模型

行动模型定义 Agent 能做什么、不能做什么、在什么条件下做、是否需要人确认。

行动分类包括：

- 查询行动：读取、检索、筛选、汇总、明细展开。
- 分析行动：计算、对比、归因、趋势分析、模拟测算。
- 解释行动：解释指标、异常、数据来源和不确定性。
- 建议行动：生成建议、备选方案、优先级和影响分析。
- 提醒行动：预警、通知、待办和关注事项。
- 协同行动：生成任务、发起确认、提交审批、请求补充数据。
- 写入行动：新增、修改、关闭、提交、回写业务系统。
- 治理行动：标注、纠错、申请权限、创建规则建议和发起数据质量工单。

行动定义模板应包括行动编码、名称、分类、风险等级、适用对象、依赖指标、依赖规则、前置条件、权限要求、人工确认要求、工具映射、输入输出、审计信息和失败处理方式。

行动模型还需要包含对象行动模型，也可以称为“对象-行动矩阵”。全局行动模型回答 Agent 具备哪些类型的动作能力，对象行动模型回答这些动作能力可以作用在哪些业务对象上。

对象行动模型建议至少定义以下内容：

| 字段 | 含义 |
| --- | --- |
| 对象类型 | 行动作用的业务对象，例如主数据对象、交易对象、资源对象、过程对象、指标对象、规则对象、数据对象 |
| 允许行动 | 该对象上允许执行的行动，例如查询、分析、解释、建议、提醒、协同、写入、治理 |
| 禁止行动 | 该对象上禁止或默认不开放的行动 |
| 风险等级 | L0 只读、L1 分析、L2 建议、L3 协同、L4 写入 |
| 前置条件 | 执行动作前必须满足的数据质量、权限、状态或规则条件 |
| 权限要求 | 哪些角色、部门或用户可以触发该动作 |
| 确认要求 | 是否需要用户确认、业务负责人确认或审批流 |
| 工具映射 | 该动作对应的数据查询、指标计算、规则判断、工单或外部 API 工具 |
| 审计要求 | 是否需要记录调用人、Agent 身份、输入、输出、使用数据和处理结果 |

示例对象行动矩阵如下：

| 对象类型 | 允许行动 | 默认限制 |
| --- | --- | --- |
| 主数据对象 | 查询、解释、质量校验、发起纠错 | 修改主数据需人工确认或审批 |
| 交易对象 | 查询、分析、状态解释、风险提醒 | 关闭、撤销、回写状态默认禁止自动执行 |
| 资源对象 | 查询、占用分析、利用率分析、预警建议 | 自动调度或分配资源需确认 |
| 指标对象 | 查询、计算、对比、解释、归因 | 修改指标口径需治理流程 |
| 规则对象 | 查询、解释、模拟命中、建议调整 | 启停规则或修改阈值需审批 |
| 数据对象 | 查询元数据、查看血缘、质量分析、发起治理工单 | 直接删除、覆盖或批量修改数据默认禁止 |

因此，行动模型不只是“Agent 有哪些工具”，还必须明确“什么对象上能用什么工具”。这能避免 Agent 把一个本应只读查询的对象误当成可修改对象，也能让权限、规则、审计和人工确认落到具体业务对象上。

### 5.7 制造业行业语义模板

前文定义的是平台级轻量语义本体元模型。对于中小型制造企业，平台还需要提供一套制造业行业语义模板，作为企业落地时的默认参考。该模板不是强制标准，而是帮助企业快速建立第一版业务对象、属性、关系、规则和 Agent 工具映射。

制造业行业语义模板可以理解为平台元模型的一个行业实例：

```text
平台级元模型：定义如何描述对象、属性、关系、规则、指标、行动和映射
制造业行业模板：定义制造企业常见对象、属性、关系、规则和工具示例
企业扩展模型：基于行业模板扩展客户自己的对象、叫法、规则和系统映射
```

#### 5.7.1 制造业核心业务对象模板

制造业模板建议先包含以下核心对象：

| 对象 | 类型 | 说明 |
| --- | --- | --- |
| Customer | 主数据对象 | 客户，用于销售、订单、发货和应收等业务 |
| Supplier | 主数据对象 | 供应商，用于采购、到货、质量和应付等业务 |
| Material | 主数据对象 / 资源对象 | 物料、料号、存货或零部件 |
| Product | 主数据对象 | 产品或成品，可关联 BOM、销售订单和生产订单 |
| BOM | 结构对象 | 产品与物料之间的组成结构 |
| SalesOrder | 交易对象 | 销售订单或客户订单 |
| SalesOrderItem | 交易对象 | 销售订单行，连接订单、产品、数量和交期 |
| PurchaseOrder | 交易对象 | 采购订单，连接供应商、物料、数量和到货计划 |
| ProductionOrder | 过程对象 / 交易对象 | 生产订单或工单，连接产品、计划、进度和产出 |
| Inventory | 资源对象 | 库存、可用量、在库量、安全库存等 |
| Shipment | 事件 / 交易对象 | 发货、出库、交付相关对象 |
| ExceptionEvent | 事件对象 | 延期、缺料、质量异常、生产异常等事件 |

这些对象不要求所有客户一次性全部启用。平台可以按客户数据成熟度逐步启用对象，并允许企业增加自己的对象类型。

#### 5.7.2 制造业典型属性模板

制造业对象的典型属性包括：

| 属性类别 | 示例 |
| --- | --- |
| 标识属性 | 编码、编号、外部系统 ID、版本 |
| 描述属性 | 名称、规格、型号、单位、分类 |
| 状态属性 | 订单状态、采购状态、生产状态、发货状态、质检状态 |
| 时间属性 | 创建时间、交期、计划开始时间、计划完工时间、实际发货时间 |
| 数量属性 | 订单数量、需求数量、库存数量、可用数量、在途数量、完工数量 |
| 规则属性 | 安全库存、最小起订量、提前期、齐套要求、质检规则 |
| 语义属性 | 是否关键物料、是否可替代、是否敏感数据、是否允许 Agent 使用 |

这些属性应作为行业模板中的推荐属性，而不是所有企业必须采用的固定字段。真实落地时，需要通过数据映射模型将这些标准属性映射到 ERP、MES、Excel、采购系统和其他来源字段。

#### 5.7.3 制造业典型关系模板

制造业对象之间的典型关系包括：

| 关系 | 含义 |
| --- | --- |
| Customer places SalesOrder | 客户下达销售订单 |
| SalesOrder contains SalesOrderItem | 销售订单包含订单行 |
| SalesOrderItem references Product | 订单行关联产品 |
| Product has_bom BOM | 产品关联 BOM |
| BOM uses Material | BOM 使用物料 |
| Material has_inventory Inventory | 物料关联库存 |
| Supplier supplies Material | 供应商供应物料 |
| PurchaseOrder purchases Material | 采购订单采购物料 |
| ProductionOrder produces Product | 生产订单生产产品 |
| ProductionOrder consumes Material | 生产订单消耗物料 |
| Shipment ships SalesOrder | 发货对象履行销售订单 |
| ExceptionEvent affects BusinessObject | 异常事件影响订单、物料、工单或发货 |

这些关系可以形成制造业业务关系图谱，帮助 Agent 从一个对象沿业务关系找到相关数据。例如，从销售订单可以追踪到订单行、产品、BOM、物料、库存、采购在途、生产订单和发货状态。

#### 5.7.4 业务关系图谱边结构

为了支持制造业对象关系图谱，关系模型需要支持“业务关系图谱边”表达。建议边结构包括：

| 字段 | 含义 |
| --- | --- |
| business_graph_edge_id | 关系边唯一标识 |
| from_entity / from_id | 起点对象类型和对象 ID |
| relation_name | 关系名称，例如 places、contains、uses、has_inventory、ships |
| to_entity / to_id | 终点对象类型和对象 ID |
| properties | 关系属性，例如数量、单位、版本、生效期、状态 |
| confidence | 关系可信度，用于区分系统事实、规则推导和 AI 建议 |
| generated_by | 关系生成方式，例如 system、mapping_rule、ai、manual |
| confirmed_by | 人工确认人或确认角色，可为空 |
| source_dataset | 关系来源数据集、表、文件或接口 |
| valid_from / valid_to | 关系生效和失效时间 |

其中，系统同步生成的关系可以直接作为事实关系；AI 推断生成的关系应标记较低可信度，并在关键场景下要求人工确认。这样既能利用 AI 辅助建模，又能避免未经确认的关系直接影响高风险决策。

#### 5.7.5 AI 工具映射示例

制造业行业模板中可以预置一组 Agent 工具映射示例，用于说明语义对象如何连接到可调用能力。

| 工具 | 作用 | 关联对象 | 默认风险等级 |
| --- | --- | --- | --- |
| get_order_detail | 查询订单详情 | SalesOrder、SalesOrderItem、Customer | L0 只读 |
| get_order_materials | 查询订单相关物料需求 | SalesOrder、Product、BOM、Material | L1 分析 |
| check_material_availability | 检查物料库存和可用性 | Material、Inventory、PurchaseOrder | L1 分析 |
| get_purchase_status | 查询采购订单和在途状态 | PurchaseOrder、Supplier、Material | L0 只读 |
| get_production_progress | 查询生产订单进度 | ProductionOrder、Product、Material | L0 只读 |
| analyze_delivery_risk | 分析交付风险 | SalesOrder、Inventory、PurchaseOrder、ProductionOrder、Shipment | L2 建议 |
| generate_daily_report | 生成经营日报或异常摘要 | Metric、BusinessRule、ExceptionEvent | L2 建议 |

工具映射不等于工具实现。它先定义“某类业务问题可以调用哪些工具、工具依赖哪些对象、风险等级是什么、是否需要人工确认”。后续实际实施时，再将工具映射到查询接口、指标服务、规则服务、报表生成服务或外部系统 API。

### 5.8 Agent 使用链路

Agent 使用语义本体层的典型链路如下：

```text
用户问题
→ 识别业务对象、属性、指标、时间和范围
→ 展开语义关系和可用路径
→ 应用规则和指标口径
→ 通过数据映射定位数据集、字段、视图或 API
→ 检查权限、质量、血缘、时效和可信等级
→ 调用查询、计算、对比、解释或建议工具
→ 输出结果、依据、口径、不确定性和建议
```

这一链路的核心是避免 Agent 直接猜测数据库结构，而是通过语义层规划数据使用路径。

### 5.9 轻量化边界

轻量语义本体层的边界如下：

- 不要求第一阶段使用完整 RDF、OWL 或三元组存储。
- 不要求业务人员维护复杂知识图谱。
- 不要求所有数据字段都进入语义层。
- 不要求所有关系都自动推理。
- 不替代数据仓库、指标系统、权限系统或工作流系统。

它首先是语义控制面和 Agent 使用数据的导航系统，后续可以根据成熟度逐步增强为知识图谱、推理服务或更复杂的语义层。

## 6. Agent 服务设计

### 6.1 Agent 定位

本平台中的 Agent 定位为基于企业可信数据的决策信息助手。

它不是自由访问所有系统的自动化机器人，也不是只基于大模型知识回答问题的通用聊天工具。它必须通过平台提供的数据、语义、业务关系图谱、规则、指标、权限和行动边界来工作。

Agent 的主要职责包括：

- 帮助用户发现和查询企业数据。
- 基于统一指标口径进行分析和对比。
- 沿业务关系图谱追踪对象之间的关联路径。
- 解释数据来源、指标计算、关系路径和异常原因。
- 根据规则生成预警和建议。
- 在需要时发起人工确认、治理工单或协同流程。
- 对回答结果提供依据、口径和不确定性说明。

Agent 服务层不直接替代 ERP、MES、采购、考勤或 OA 等业务系统。它优先作为企业数据和业务语义之上的决策辅助层，帮助用户更快理解情况、定位原因、形成建议，并在受控条件下进入协同或写入流程。

### 6.2 Agent 问题处理链路

Agent 处理企业业务问题时，应遵循受控链路，而不是直接把用户问题翻译成 SQL 或系统操作。

典型链路如下：

```text
用户问题
→ 意图识别
→ 业务对象识别
→ 时间、组织、权限和范围识别
→ 关系路径规划
→ 数据映射定位
→ 数据质量和可信度检查
→ 指标计算或规则判断
→ 工具调用
→ 回答生成
→ 依据、口径、不确定性和审计记录
```

各环节的作用如下：

| 环节 | 作用 |
| --- | --- |
| 意图识别 | 判断用户是在查询、分析、解释、预警、生成建议、发起协同还是请求写入 |
| 业务对象识别 | 将问题映射到 Customer、SalesOrder、Material、Inventory、ProductionOrder 等语义对象 |
| 范围识别 | 识别时间范围、组织范围、客户范围、产品范围、权限范围和数据敏感等级 |
| 关系路径规划 | 基于 SemanticRelation 和 BusinessGraphEdge 判断应沿哪些对象关系查找数据 |
| 数据映射定位 | 通过 DataMapping 找到实际表、字段、视图、文件、API 或指标服务 |
| 质量可信检查 | 检查数据来源、更新时间、质量状态、冲突情况、可信等级和血缘 |
| 指标规则计算 | 按统一口径计算指标，按规则模型执行异常判断、风险判断和解释规则 |
| 工具调用 | 调用查询、分析、规则、报告、协同或写入工具 |
| 回答生成 | 输出结论、证据、口径、来源、关系路径、不确定性和可选建议 |
| 审计记录 | 记录用户、Agent、问题、数据、工具、规则、输出和确认结果 |

这一链路的核心目标是让 Agent 的每个回答都有路径、有依据、有边界，而不是依赖模型自由猜测。

### 6.3 Agent 与业务关系图谱

业务关系图谱是 Agent 使用企业数据的导航结构。

在制造企业中，很多问题不是单表查询，而是跨对象追踪。例如用户问“这个客户订单有没有交付风险”，Agent 需要从销售订单出发，沿业务关系找到订单行、产品、BOM、物料、库存、采购在途、生产订单和发货状态。

典型路径如下：

```text
SalesOrder
→ SalesOrderItem
→ Product
→ BOM
→ Material
→ Inventory
→ PurchaseOrder
→ ProductionOrder
→ Shipment
→ ExceptionEvent
```

Agent 使用业务关系图谱时，需要区分两类信息：

| 类型 | 说明 | 示例 |
| --- | --- | --- |
| 关系类型 | 由 SemanticRelation 定义，说明对象类型之间允许存在什么关系 | SalesOrder contains SalesOrderItem |
| 关系实例 | 由 BusinessGraphEdge 记录，说明具体对象实例之间实际存在什么关系 | SO-001 contains SO-001-Item-01 |

因此，Agent 的查询过程不是“猜表名和字段”，而是：

```text
识别核心对象
→ 查找可用关系类型
→ 沿业务关系图谱边展开相关对象
→ 根据数据映射找到真实数据
→ 调用工具完成查询、计算或判断
```

业务关系图谱还可以帮助 Agent 做解释。例如，当 Agent 判断某订单存在延期风险时，可以说明风险来自哪条路径：是关键物料库存不足、采购到货延期、生产工单未开工，还是发货状态异常。

### 6.4 Agent 工具调用机制

Agent 不应直接访问数据库或业务系统，而应通过受控工具调用平台能力。

可用工具类型包括：

| 工具类型 | 作用 | 示例 |
| --- | --- | --- |
| 语义检索工具 | 查询对象、属性、关系、指标和数据映射 | 查找“交付风险”依赖哪些对象和规则 |
| 关系图谱工具 | 查询对象实例之间的业务关系路径 | 查询订单关联的产品、BOM、物料和库存 |
| 数据查询工具 | 查询数据集、视图、明细和聚合结果 | 查询某物料当前库存和在途采购 |
| 指标计算工具 | 按统一口径计算指标 | 计算订单准交率、库存周转率、产能利用率 |
| 规则判断工具 | 执行质量规则、业务规则、权限规则和解释规则 | 判断是否缺料、是否逾期、是否齐套 |
| 报告生成工具 | 生成分析摘要、异常说明和管理报告 | 生成每日交付风险报告 |
| 协同工具 | 创建数据治理工单、请求业务确认或提交审批 | 发起物料主数据纠错工单 |
| 写入工具 | 在确认或审批后回写业务系统 | 回写状态、关闭任务、提交审批结果 |

每个工具都应定义：

- 工具名称和业务用途。
- 关联对象、关联指标和关联规则。
- 输入参数、输出格式和错误返回。
- 可调用角色、权限要求和数据范围。
- 默认风险等级和人工确认要求。
- 数据来源、质量要求和可信等级。
- 调用日志、审计要求和失败处理方式。

工具调用前，Agent 需要完成权限、对象行动策略和数据质量检查。工具调用后，Agent 需要将工具输出转化为业务语言，并保留可追溯依据。

### 6.5 Agent 回答可信机制

Agent 的回答必须能被业务人员追问、核验和复盘。

平台应要求 Agent 在回答中尽量包含以下信息：

| 信息 | 说明 |
| --- | --- |
| 结论 | 对用户问题的直接回答 |
| 关键依据 | 使用了哪些数据、指标、规则或关系路径 |
| 数据来源 | 来源系统、数据集、表、文件、接口或同步批次 |
| 时间范围 | 数据统计或查询覆盖的时间 |
| 指标口径 | 指标公式、过滤条件、维度和口径版本 |
| 关系路径 | 从核心对象追踪到相关对象的路径 |
| 数据质量 | 数据是否完整、是否过期、是否存在冲突或缺失 |
| 不确定性 | 哪些结论可信，哪些需要人工确认或补充数据 |
| 建议动作 | 可选建议、风险等级和是否需要确认 |

对于不同类型问题，可信回答的重点不同：

| 问题类型 | 回答重点 |
| --- | --- |
| 查询类 | 明确结果、来源、更新时间和权限范围 |
| 分析类 | 明确指标口径、对比维度、过滤条件和样本范围 |
| 解释类 | 明确异常原因、影响对象、规则依据和关系路径 |
| 建议类 | 明确建议依据、预期影响、风险等级和替代方案 |
| 协同类 | 明确需要谁确认、确认什么、后续动作是什么 |
| 写入类 | 明确写入对象、变更内容、审批状态和审计记录 |

当数据不足、口径不明确、关系路径不完整或权限受限时，Agent 应说明限制，而不是输出确定性结论。

### 6.6 Agent 风险控制

Agent 能力按风险等级分层。

| 等级 | 能力 | 默认策略 |
| --- | --- | --- |
| L0 只读 | 查询、检索、查看 | 默认允许，但受权限控制 |
| L1 分析 | 计算、对比、解释 | 默认允许，但需引用指标口径和数据来源 |
| L2 建议 | 生成建议、预警、待办草稿 | 默认允许，但必须标明建议依据和不确定性 |
| L3 协同 | 发起确认、提交审批、创建任务 | 需要用户确认 |
| L4 写入 | 修改数据、回写系统、触发业务动作 | 默认禁止自动执行，需审批和审计 |

这种分级方式可以让平台先释放查询、分析和解释价值，同时避免 Agent 过早介入高风险业务动作。

Agent 风险控制包括五个方面。

第一，权限控制。

Agent 只能在当前用户权限范围内访问数据。敏感字段、跨部门数据和高风险数据必须进行权限校验和必要脱敏。

第二，数据质量控制。

Agent 使用数据前需要检查质量状态、更新时间、来源可信度和冲突情况。质量不足时，应提示用户而不是给出确定性结论。

第三，口径控制。

涉及指标和经营分析时，Agent 必须使用平台定义的指标口径，并在回答中说明口径、时间范围和过滤条件。

第四，行动控制。

涉及协同、审批、写入、关闭、回写等动作时，必须根据行动等级进行人工确认和审计。

第五，解释控制。

Agent 输出结果时，应尽量说明使用的数据源、关键规则、指标口径、时间范围和不确定性。对于无法确认的问题，应明确说明数据不足或口径不确定。

除了上述控制，还需要结合对象行动策略进行约束。即使某个用户有权查看某类数据，也不代表 Agent 可以对该对象执行写入、关闭、审批或回写动作。对象行动策略应作为 Agent 工具调用前的强制检查条件。

## 7. 能力成熟度与落地边界

### 7.1 可信数据底座

可信数据底座是平台的第一层成熟能力。

目标是解决“企业数据能不能接进来、管起来、查得到”的问题。

核心能力包括：

- 多源数据接入：ERP、MES、采购、考勤、Excel、共享盘、数据库和 API。
- Raw 层保真：保留原始数据、来源系统、同步批次和审计信息。
- Clean 层轻治理：进行校验、去重、轻标准化和质量标记。
- 基础数据目录：让数据集、字段、来源和同步状态可发现。
- 权限控制：建立基础角色、数据集权限、字段敏感级别和访问审计。
- 基础查询服务：让用户和 Agent 能在权限范围内查询可信数据。

该阶段的关键判断标准是：数据是否能稳定接入，数据来源是否清楚，数据质量是否可见，基础查询是否可用。

### 7.2 语义治理与指标分析

语义治理与指标分析是平台的第二层成熟能力。

目标是解决“数据能不能被业务一致理解”的问题。

核心能力包括：

- 业务对象模型：定义客户、供应商、物料、产品、订单、工单、库存、设备、员工等核心对象。
- 属性与关系模型：定义对象属性、对象之间的关系和关系路径。
- 数据映射模型：将语义对象、属性、关系和指标映射到真实表、字段、文件和 API。
- 规则模型：定义质量规则、业务判断规则、权限规则和解释规则。
- 指标模型：统一经营指标口径、维度、公式、依赖数据和可信等级。
- 语义目录：让业务人员和 Agent 可以理解数据资产的业务含义。

该阶段的关键判断标准是：指标是否有统一口径，业务对象是否有清晰定义，Agent 是否能通过语义层找到正确数据并解释结果。

### 7.3 Agent 决策辅助闭环

Agent 决策辅助闭环是平台的第三层成熟能力。

目标是解决“Agent 能不能安全参与业务决策”的问题。

核心能力包括：

- Agent 查询与分析：基于语义层进行自然语言问答、查询规划和指标分析。
- 异常解释与归因：结合规则、指标、关系和数据血缘解释异常原因。
- 预警与建议：基于规则和指标生成风险提示、建议动作和影响说明。
- 行动模型：定义 Agent 可执行动作、风险等级、权限要求和人工确认条件。
- 协同闭环：将建议转化为待办、工单、审批或人工确认流程。
- 审计反馈：记录问题、数据、规则、工具调用、回答、确认和处理结果。

该阶段的关键判断标准是：Agent 的建议是否有依据，行动是否可控，协同是否闭环，审计是否完整。

### 7.4 不做什么

为保持平台初期方案可落地，需要明确不做什么。

第一，不一开始做万能 Agent。

Agent 必须依赖平台数据、语义、规则和权限工作，而不是自由访问所有系统和文件。

第二，不一开始做重型知识图谱。

平台先做轻量语义元模型和数据映射。知识图谱和复杂推理能力可以作为后续增强，而不是初期前提。

第三，不一开始要求所有数据完全标准化。

对于来源复杂、质量不稳的数据，应先接入、保留来源、标记质量，再逐步治理。

第四，不默认自动写回业务系统。

涉及 ERP、MES、采购、考勤、财务等核心系统的数据修改、状态变更和业务动作，必须经过权限校验、人工确认和审计。

第五，不替代企业原有业务系统。

平台是数据汇总、治理、语义建模和 Agent 服务平台，不是 ERP、MES、采购、考勤或财务系统的替代品。

### 8.3 参考方法论

本方案参考以下方法论，但不照搬复杂实现：

- 分层数据架构：用于组织 Raw、Clean、Business、Analytics 数据生命周期。
- 维度建模：用于组织事实、维度和指标分析。
- 语义层：用于统一指标、实体、维度和业务口径。
- 本体建模：用于借鉴对象、属性、关系、约束和规则表达思想。
- 数据血缘：用于追踪数据来源、加工过程、上下游影响和审计路径。
- AI 风险治理：用于设计 Agent 权限、行动分级、人工确认和审计机制。

### 8.4 后续待确认事项

后续在进入更详细方案或实施阶段前，需要进一步确认：

- 第一批客户常见 ERP、MES、采购、考勤系统类型。
- 典型 Excel 模板和共享盘文件类型。
- 制造业核心业务对象的标准清单。
- 第一批高价值指标清单。
- Agent 初期允许开放的能力范围。
- 客户对本地部署、私有化部署或云部署的要求。
- 数据安全、脱敏、权限和审计的企业要求。
- 是否需要对接现有 BI、报表、工作流或企业 IM 系统。
