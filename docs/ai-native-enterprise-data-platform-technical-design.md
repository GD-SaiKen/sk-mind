## 1. 技术设计定位

本技术设计方案回答研发团队在进入实现前必须明确的问题：

- 平台当前由哪些技术模块组成。
- 哪些能力放在同一个后端应用中，哪些能力通过异步任务执行。
- 数据从 ERP、MES、Excel、共享盘、数据库和 API 接入后如何落库、分层、治理和开放给 Agent。
- 语义模型、业务关系图谱和 Agent 工具调用如何用轻量方式实现。
- 权限、脱敏、审计、质量状态和来源追踪如何贯穿全链路。
- 当前推荐使用哪些技术栈，哪些复杂技术不纳入当前方案。

本文档不直接给出完整数据库 DDL、完整 API 文档或研发排期。它给出的是足够进入研发拆解的架构边界和设计方向。研发实施时，可以在此基础上继续拆分数据库设计、接口设计、任务清单、测试用例和部署手册。

## 2. 设计原则与复杂度约束

### 2.1 面向中小型制造企业

目标客户通常具备以下特征：

- 有 ERP、MES、采购、仓储、考勤、财务等部分业务系统，但系统开放能力不一致。
- 大量历史数据仍存在 Excel、本地文件夹、共享盘和部门台账中。
- 客户现场 IT 运维能力有限，不适合部署复杂集群。
- 企业最需要的是数据接得进来、查得到、来源说得清、质量看得见。
- Agent 应优先做查询、分析、解释和建议，不默认写回核心业务系统。

因此，技术方案必须优先保证私有化部署简单、现场问题容易排查、基础能力可稳定交付。

### 2.2 简单高效可用

当前架构采用以下约束：

- 后端采用模块化单体，而不是微服务。
- 数据存储采用 PostgreSQL，而不是湖仓、数据湖或多种专用存储组合。
- 业务关系图谱采用关系表和索引，而不是独立图数据库。
- 语义模型采用关系模型和 JSONB 灵活字段，而不是 RDF、OWL 或三元组存储。
- 异步任务采用轻量队列，而不是 Airflow、Spark 或复杂调度平台。
- Excel 解析采用规则、模板和人工确认，而不是依赖 AI 自动理解所有文件。
- Agent 通过受控工具使用数据，而不是直接生成 SQL 访问任意数据库。

简单不是能力弱，而是把技术复杂度控制在客户现场能部署、能维护、能排查的范围内。

### 2.3 当前不纳入的复杂技术

当前方案不纳入以下复杂技术：

- 微服务网关、服务注册中心和复杂服务治理。
- Hadoop、Spark、Flink、湖仓体系和大型数据平台套件。
- 独立知识图谱平台、RDF、OWL、三元组存储和复杂推理机。
- Kubernetes 作为私有化部署必选项。
- 大规模实时流处理平台。
- 让 Agent 直接访问业务数据库、共享盘或任意业务系统接口。

这些技术会增加部署、运维、排错和研发协作成本，不符合当前产品服务中小制造企业的简单高效原则。

## 3. 推荐技术栈

| 层级 | 推荐技术 | 选择理由 |
| --- | --- | --- |
| 前端 | Vue 3 + Vite + TypeScript | 适合管理控制台、表单、列表、状态页和复杂配置页 |
| UI 组件 | Element Plus | Vue 生态成熟组件库，适合企业后台、表格、表单、弹窗和权限态 |
| 后端 API | Python + FastAPI | 开发效率高，适合数据接入、AI 工具编排和管理 API |
| ORM 与迁移 | SQLAlchemy + Alembic | 适合 PostgreSQL 关系模型和持续维护 |
| 数据库 | PostgreSQL | 同时承载业务数据、元数据、JSONB、索引和基础分析查询 |
| 异步任务 | RQ + Redis | 轻量，适合私有化部署中的导入、质量检查和图谱生成任务 |
| 定时触发 | APScheduler 或系统 cron 触发后端任务 | 满足简单定时同步和定时检查 |
| Excel 解析 | openpyxl，必要时补充 pandas | openpyxl 适合确定性解析，pandas 用于样本预览和轻量转换 |
| 文件处理 | Python 标准库 + pathlib + hashlib | 支持目录扫描、文件 hash、增量识别和审计 |
| API 文档 | OpenAPI | FastAPI 自动生成，便于前后端和实施团队协作 |
| 反向代理 | Nginx | 统一前端静态资源、API 代理、HTTPS 和上传大小控制 |
| 部署 | Docker Compose | 适合客户内网单机或小型服务器私有化部署 |
| 日志 | Python logging + 结构化 JSON 日志 | 简单可查，便于现场排错 |
| 监控 | 健康检查接口 + 任务状态页 + 日志文件 | 满足现场运维可见性 |

技术栈选择原则是成熟、易部署、易排查、团队容易掌握。当前方案不追求技术栈炫技，也不为少量场景引入大型平台依赖。

## 4. 总体技术架构

### 4.1 架构形态

平台当前采用“模块化单体 + 异步任务 + PostgreSQL 优先”的架构。

```text
浏览器管理控制台
  -> Nginx
  -> FastAPI 后端应用
       -> 用户、权限、审计模块
       -> 数据源与接入配置模块
       -> 数据目录与元数据模块
       -> 数据质量模块
       -> 轻量语义模型模块
       -> 业务关系图谱模块
       -> 指标与规则模块
       -> Agent 数据服务模块
       -> 任务调度与任务状态模块
  -> Redis 队列
  -> 后台 Worker
       -> 文件扫描
       -> Excel 解析
       -> 数据库/API 同步
       -> 质量检查
       -> Clean/Serving 生成
       -> 关系边生成
       -> Agent 报告生成
  -> PostgreSQL
       -> 平台业务库
       -> Raw/Clean/Serving 数据层
       -> 元数据与语义模型
       -> 权限与审计日志
```

### 4.2 三条核心链路

平台技术架构围绕三条链路组织。

| 链路 | 技术目标 | 关键模块 |
| --- | --- | --- |
| 数据生产链路 | 将数据源接入平台，形成可追踪、可治理、可查询的数据资产 | 数据源、接入任务、Raw、Clean、Serving、质量、目录 |
| 语义治理链路 | 将表字段转化为业务对象、关系、规则、指标和 Agent 行动边界 | 业务对象、属性、关系、映射、指标、规则、对象行动策略 |
| Agent 消费链路 | 让 Agent 在权限和质量约束下查询、分析、解释和建议 | 语义检索、工具注册、权限检查、查询服务、回答生成、审计 |

### 4.3 模块化单体边界

模块化单体不是把所有代码写在一起，而是在一个后端进程中保持清晰模块边界：

- 每个模块有独立的数据模型、服务层、API 路由和权限点。
- 模块之间通过服务接口调用，不直接跨模块修改内部数据。
- 长时间任务统一进入任务队列，由 Worker 执行。
- Worker 任务调用服务层能力，不绕过业务规则直接写库。
- Agent 工具必须通过工具注册和权限检查，不直接访问任意表。

这种方式可以避免当前阶段承受微服务部署、网络调用、分布式事务和链路追踪复杂度。

## 5. 私有化部署架构

### 5.1 默认部署形态

平台默认部署在客户内网的一台应用服务器或小型虚拟机上。

```text
客户内网
├─ 用户浏览器
├─ AI 原生企业数据平台服务器
│  ├─ Nginx
│  ├─ 前端静态资源
│  ├─ FastAPI 后端
│  ├─ Worker 进程
│  ├─ Redis
│  └─ PostgreSQL
├─ ERP / MES / 采购 / 考勤等业务系统
├─ 业务系统数据库
└─ 共享盘 / 文件服务器 / 本地目录
```

该部署方式的目标是降低现场交付难度。平台通过内网访问业务数据库、API、共享盘和文件目录，业务数据默认不离开客户环境。

### 5.2 推荐资源配置

| 环境 | CPU | 内存 | 存储 | 说明 |
| --- | --- | --- | --- | --- |
| 试点环境 | 4 核 | 8 GB | 200 GB SSD | 支持少量数据源、Excel 导入和基础 Agent 查询 |
| 生产起步 | 8 核 | 16 GB | 500 GB SSD | 支持核心业务数据、定时任务和多用户使用 |
| 数据较多客户 | 16 核 | 32 GB | 1 TB SSD 起 | 用于更大的 Excel 文件规模和更多同步任务 |

资源配置按数据量、Excel 文件规模、同步频率和并发用户数调整，但部署形态仍保持简单。

### 5.3 网络与安全边界

私有化部署需要遵循以下网络原则：

- 平台访问业务系统时优先使用只读账号。
- 数据库直连应限制访问库表和来源 IP。
- API 接入密钥在平台内加密保存，页面只展示摘要。
- 共享盘访问由后端服务完成，浏览器不直接读取客户本地目录。
- 如果使用外部大模型 API，必须由客户明确批准数据出网范围。
- 默认策略是不将明细业务数据发送到客户外部环境。

### 5.4 备份与恢复

当前方案至少需要支持：

- PostgreSQL 定时备份。
- 配置、元数据、语义模型、任务记录和审计日志备份。
- 文件导入记录和错误报告可恢复。
- 版本升级前自动备份数据库。
- 恢复后能重新查看任务批次、来源、质量状态和 Agent 审计记录。

### 6.1 模块总览

| 模块 | 主要职责 |

| 用户与权限模块 | 用户、角色、部门、数据权限、字段权限和脱敏策略 |
| 数据源模块 | 数据源登记、连接配置、负责人、接入方式和状态管理 |
| 接入任务模块 | 任务配置、执行触发、批次记录、错误清单和重试 |
| 文件与 Excel 模块 | 目录扫描、文件 hash、样本预览、模板规则和批量解析 |
| 数据集模块 | Raw、Clean、Serving 数据集和字段结构管理 |
| 数据目录模块 | 数据集目录、字段目录、来源、负责人、标签和 Agent 可用性 |
| 数据质量模块 | 质量规则、执行结果、问题清单和质量状态 |
| 血缘模块 | 来源到 Raw/Clean/Serving/语义映射的链路记录 |
| 语义模型模块 | BusinessObject、ObjectProperty、SemanticRelation、DataMapping |
| 业务关系图谱模块 | BusinessGraphEdge 的生成、查询、确认和可信度管理 |
| 指标与规则模块 | Metric、BusinessRule、口径、规则判断和解释模板 |
| Agent 服务模块 | 问题处理、工具注册、工具调用、回答生成和审计 |
| 审计模块 | 用户访问、数据查询、权限变更、Agent 工具调用记录 |
| 系统设置模块 | 字典、标签、状态、参数和运行配置 |

### 6.2 后端分层

后端代码建议按以下结构组织：

```text
app/
  api/                 # FastAPI 路由
  modules/
    auth/
    data_sources/
    ingestion/
    datasets/
    catalog/
    quality/
    lineage/
    semantic/
    graph/
    metrics/
    agent/
    audit/
    settings/
  workers/             # RQ 任务入口
  core/                # 配置、数据库、日志、权限依赖
  tests/
```

分层原则：

- API 层只处理请求参数、鉴权、响应格式和错误码。
- 服务层处理业务规则、状态流转和跨模块编排。
- 数据访问层封装 SQLAlchemy 查询和事务。
- Worker 任务调用服务层。
- Agent 工具调用受服务层约束。

## 7. 数据接入与异步任务架构

### 7.1 接入方式

当前支持以下接入方式：

| 接入方式 | 技术实现方向 | 重点 |
| --- | --- | --- |
| Excel 导入 | openpyxl 读取 workbook、sheet、行列和单元格 | 样本确认、模板规则、批量导入 |
| CSV 导入 | Python csv 或 pandas 轻量处理 | 文件编码、分隔符、表头识别 |
| 共享盘扫描 | 后端按目录路径扫描，记录文件元数据和 hash | 增量识别、失败文件清单 |
| 数据库同步 | SQLAlchemy 或数据库驱动只读连接 | 指定表、字段范围、增量字段 |
| API 拉取 | requests/httpx 调用接口 | 分页、鉴权、重试、错误记录 |
| 手工录入 | 管理台表单保存配置或少量数据 | 受控、审计、适合配置类数据 |

CDC、实时流处理和复杂数据库日志解析不纳入当前接入能力。

### 7.2 任务执行链路

所有长时间接入和处理任务进入异步队列。

```text
用户创建任务
→ 后端保存任务配置
→ 后端创建任务批次
→ 投递 Redis 队列
→ Worker 拉取任务
→ 执行扫描、解析、同步或质量检查
→ 分批写入 PostgreSQL
→ 更新批次状态、数据量和错误明细
→ 刷新数据目录、质量状态和血缘记录
```

任务状态建议包括：

- draft：草稿。
- pending：待执行。
- running：运行中。
- success：成功。
- partial_success：部分成功。
- failed：失败。
- canceled：已取消。
- disabled：已停用。

### 7.3 批次与错误处理

每次任务执行都生成独立批次。批次必须记录：

- 任务 ID。
- 执行人或触发来源。
- 开始时间和结束时间。
- 输入文件数、记录数、成功数、失败数和跳过数。
- 产出数据集或数据表。
- 错误类型、错误位置和错误原因。
- 是否影响 Agent 可用性。

失败处理遵循以下原则：

- 单个文件失败不阻塞其他文件导入。
- 单行数据失败不阻塞整个文件，除非模板规则不可识别。
- 结构变化、字段缺失和权限失败必须进入错误清单。
- 重试生成新批次，不覆盖旧批次记录。
- 用户可以按批次追溯数据来源和处理结果。

### 7.4 调度策略

当前采用轻量调度：

- 手动执行：用户在管理台点击执行或重试。
- 简单定时：后端定时器或系统 cron 定期触发任务。
- 按需执行：Agent 或用户请求触发受控查询或报告生成。

任务调度重点是可配置、可查看、可重试、可审计，不引入复杂 DAG 平台。

## 8. 平台数据库与数据分层设计

### 8.1 PostgreSQL 优先

平台数据库统一采用 PostgreSQL，承载：

- 平台配置。
- 用户、权限和审计。
- 数据源、任务和批次。
- Raw、Clean、Serving 数据层。
- 数据目录、字段元数据、质量结果和血缘。
- 语义模型、指标、规则和数据映射。
- 业务关系图谱边。
- Agent 会话、工具调用和回答审计。

PostgreSQL 的优势是私有化部署简单、关系模型表达能力强、JSONB 字段灵活、事务和索引可靠，适合当前中小制造企业场景。

### 8.2 数据分层

当前采用四层逻辑分层，物理上都在 PostgreSQL 中实现。

| 层级 | 技术形态 | 主要作用 |
| --- | --- | --- |
| Raw 层 | 原始表、raw_record、JSONB 或导入结果表 | 保留原始数据、来源、批次和文件行号 |
| Clean 层 | 清洗后表、标准化字段、质量标记 | 类型转换、格式标准化、去重和质量状态 |
| Serving 层 | 稳定视图或服务表 | 面向用户查询、数据目录、语义映射和 Agent 工具 |
| Semantic 层 | 元数据表和映射表 | 描述业务对象、属性、关系、规则、指标和行动边界 |

这里的分层是数据治理分层，不是独立数据湖或数仓集群。

### 8.3 Raw 层设计

Raw 层必须保留来源和批次。建议采用两种方式组合：

- 对结构稳定的数据源，生成对应 Raw 物理表。
- 对结构不稳定或探索中的数据，使用 raw_record 表加 JSONB 保存原始行。

Raw 层每条记录至少需要追踪：

- 数据源 ID。
- 接入任务 ID。
- 批次 ID。
- 来源系统、来源表、来源文件或来源 API。
- 文件路径、文件名、sheet 名和行号。
- 原始字段和值。
- 导入时间。
- 原始 hash 或行 hash。
- 原始校验状态。

Raw 层不做强业务统一，目标是保真、可追溯、可重放。

### 8.4 Clean 层设计

Clean 层负责轻标准化和质量标记。

Clean 层处理内容包括：

- 字段类型转换。
- 日期、金额、数量、单位和编码格式标准化。
- 必填、唯一、枚举、格式和类型校验。
- 去重或重复标记。
- 质量状态和质量问题关联。
- 来源批次、来源 Raw 记录和处理规则版本。

Clean 层不强行修正无法确认的业务冲突。对于编码冲突、口径冲突和跨系统差异，应先标记问题，再进入业务确认或治理流程。

### 8.5 Serving 层设计

Serving 层是面向用户查询、Agent 工具和语义映射的稳定入口。

Serving 层可以采用：

- SQL View。
- Materialized View。
- 由任务生成的服务表。
- 查询服务封装的受控数据集。

Agent 不直接访问未知 Raw 表。Agent 工具优先访问 Serving 层、指标服务、语义映射后的数据集或经过权限封装的查询接口。

## 10. 元数据、目录、质量与血缘设计

### 10.1 数据目录

数据目录不是简单表清单，而是人和 Agent 理解平台数据资产的入口。

数据目录至少展示：

- 数据集名称、类型、层级和状态。
- 来源系统、接入任务和最近批次。
- 字段列表、字段类型、业务说明和样例值。
- 负责人、业务域、标签和更新时间。
- 质量状态、问题数量和最近检查时间。
- 权限等级、敏感字段和脱敏策略。
- 是否允许 Agent 查询。
- 关联语义对象和数据映射。

### 10.2 质量检查

当前质量检查以基础规则为主：

- 空值检查。
- 唯一性检查。
- 类型检查。
- 日期、金额、编码格式检查。
- 枚举值检查。
- 重复文件和重复记录检查。
- 数据更新时间检查。
- 简单关联完整性检查。

质量检查结果必须能被用户和 Agent 读取。Agent 回答时，如果所用数据存在质量问题，应在回答中说明限制。

### 10.3 轻量血缘

当前血缘不追求复杂图谱化展示，重点记录关键链路：

```text
数据源 / 文件 / API / 来源表
→ 接入任务
→ 批次
→ Raw 数据集
→ Clean 数据集
→ Serving 数据集
→ 语义映射 / 指标 / Agent 工具
```

血缘记录用于：

- 追踪数据来源。
- 分析字段变化影响。
- 解释 Agent 回答依据。
- 定位质量问题来自哪个批次或文件。
- 支持审计复盘。

## 11. 轻量语义模型技术设计

### 11.1 实现方式

轻量语义模型使用 PostgreSQL 关系表实现。

核心做法：

- 业务对象、属性、关系、指标、规则和行动策略使用结构化表保存。
- 灵活配置使用 JSONB 保存，例如别名、枚举映射、规则参数和工具参数。
- 语义对象与真实数据资产通过 data_mapping 建立连接。
- 语义检索采用关键词、标签、别名和同义词检索。

当前不采用 RDF、OWL、三元组存储和复杂推理机。

### 11.2 语义对象管理

业务对象用于定义企业业务世界中的核心对象，例如 Customer、Supplier、Material、Product、SalesOrder、ProductionOrder、Inventory。

业务对象至少需要记录：

- 对象编码和名称。
- 对象分类。
- 业务定义和适用范围。
- 别名和同义词。
- 主标识和候选标识。
- 负责人和可信等级。
- 权限等级和 Agent 可用性。
- 生命周期状态。

### 11.3 语义映射

DataMapping 负责把语义对象和真实数据连接起来。

映射类型包括：

- 对象映射：业务对象到数据集、表、文件或 API。
- 属性映射：对象属性到字段。
- 关系映射：对象关系到关联键、外键、中间表或生成规则。
- 指标映射：指标到数据集、字段、公式和过滤条件。

映射必须记录可信等级和确认状态。系统生成或 AI 辅助生成的映射，不能默认等同于人工确认映射。

### 11.4 规则和指标

规则和指标以配置化表达为主，不引入复杂规则引擎。

可采用：

- 固定规则类型加参数。
- SQL 模板加受控参数。
- Python 内置函数白名单。
- 指标公式表达式白名单。

不允许用户或 Agent 直接提交任意 SQL 或任意 Python 代码执行。

## 12. 业务关系图谱轻量实现

### 12.1 实现原则

业务关系图谱使用 PostgreSQL 表实现，不单独引入图数据库。

适用原因：

- 当前关系深度有限。
- 制造业核心对象数量可控。
- 关系边可以通过索引支持常见查询。
- 私有化部署更简单。
- 研发和运维复杂度更低。

### 12.2 关系边生成

关系边来源包括：

- 数据库外键或关联键。
- Excel 模板字段。
- 业务规则生成。
- 人工维护。
- AI 辅助建议后人工确认。

关键关系边必须记录生成方式和可信度。AI 生成的关系默认需要人工确认后才能进入高可信图谱。

### 12.3 查询边界

关系查询建议限制：

- 默认查询深度不超过 3 层。
- 查询必须指定起点对象或对象类型。
- 关系类型使用白名单。
- 返回结果数量有上限。
- Agent 查询关系路径前必须检查对象权限和数据可用性。

这些限制能保证图谱查询简单、可控、可解释，不把当前产品变成复杂图计算平台。

## 13. Agent 数据服务与工具调用链路

### 13.1 Agent 定位

Agent 是基于平台可信数据和轻量语义控制面的决策信息助手，不是自由访问所有系统的自动化机器人。

当前 Agent 重点开放：

- L0 只读查询。
- L1 分析和解释。
- L2 建议和预警草稿。

L3 协同需要用户确认。L4 写入默认关闭，不作为当前自动能力。

### 13.2 技术链路

Agent 处理问题时遵循受控链路：

```text
用户问题
→ 记录会话和用户身份
→ 意图识别
→ 识别业务对象、指标、时间和范围
→ 查询语义模型和数据目录
→ 规划关系路径和数据映射
→ 检查用户权限、对象行动策略、数据质量和时效
→ 选择受控工具
→ 执行查询、指标计算、规则判断或关系图谱查询
→ 汇总工具结果
→ 生成回答、依据、不确定性和建议
→ 保存 Agent 审计记录
```

### 13.3 工具注册

Agent 工具必须注册后才能调用。每个工具需要定义：

- 工具编码和名称。
- 工具类型。
- 适用对象。
- 输入参数 schema。
- 输出结果 schema。
- 权限要求。
- 风险等级。
- 数据质量要求。
- 是否需要人工确认。
- 审计要求。

工具类型包括：

- 数据目录查询工具。
- 数据集查询工具。
- 指标计算工具。
- 质量状态查询工具。
- 业务关系图谱查询工具。
- 规则判断工具。
- 报告生成工具。
- 治理工单工具。

### 13.4 查询安全

Agent 不直接执行任意 SQL。建议采用以下方式：

- Agent 输出结构化查询意图，而不是 SQL 字符串。
- 后端根据工具定义和数据映射生成受控查询。
- 查询只能访问允许 Agent 使用的数据集或 Serving 层视图。
- 查询字段必须经过权限和敏感字段检查。
- 返回数据量必须限制。
- 所有工具调用都保存输入、输出摘要、使用数据和执行状态。

### 13.5 大模型适配

私有化部署下，大模型能力通过 LLM Adapter 封装。

LLM Adapter 应支持：

- 本地模型服务。
- 客户批准的私有模型 API。
- 客户批准的外部模型 API。

默认原则：

- 不把明细业务数据发送到客户外部环境。
- 如果需要外部模型处理，必须经过客户安全审批和配置开关。
- Prompt 中优先使用结构化元数据、字段说明、规则摘要和必要结果摘要，而不是大批量原始明细。
- 所有模型调用记录调用时间、用户、用途、输入摘要和输出摘要。

## 14. 权限、安全、脱敏与审计

### 14.1 权限模型

当前采用 RBAC 加数据权限组合：

- 用户。
- 角色。
- 部门。
- 数据集权限。
- 字段敏感等级。
- Agent 工具权限。
- 对象行动策略。

权限判断必须同时作用于用户页面访问和 Agent 工具调用。Agent 不能绕过当前用户权限。

### 14.2 敏感字段与脱敏

字段可标记为：

- 普通。
- 内部。
- 敏感。
- 高敏。

脱敏策略包括：

- 完全隐藏。
- 部分展示。
- 聚合后展示。
- 仅管理员或指定角色可见。

Agent 回答中不得泄露用户无权查看的敏感字段。对于权限不足的问题，Agent 应说明无法访问相关数据，而不是猜测答案。

### 14.3 审计日志

审计日志至少覆盖：

- 登录和退出。
- 数据源配置变更。
- 接入任务执行和重试。
- 数据集查询和导出。
- 字段说明、权限、敏感等级变更。
- 质量规则执行和问题处理。
- 语义对象、映射、规则、指标变更。
- Agent 会话、工具调用、回答依据和失败原因。
- 人工确认和审批记录。

审计日志应可按用户、时间、数据集、工具、对象和操作类型查询。

## 18. 技术验收标准

- Clean 层可以执行基础校验和质量标记。
- Serving 层可以提供受控查询入口。
- 数据目录可以展示数据集、字段、来源、质量、权限和 Agent 可用性。
- 权限和敏感字段脱敏对用户页面和 Agent 工具同时生效。
- 可以维护业务对象、属性、关系和数据映射。
- 可以生成或维护 BusinessGraphEdge，并记录来源和可信度。
- Agent 可以通过受控工具查询数据目录、数据集、质量状态和关系路径。
- Agent 回答能够记录数据来源、工具调用、质量提示和不确定性。
- 高风险行动默认不自动执行。
- 审计日志可以追踪用户访问、任务执行、权限变更和 Agent 工具调用。

## 19. 总结

本技术方案的核心取向是：用简单、可部署、可维护的架构，把 AI 原生企业数据平台从产品方案推进到研发可落地状态。

当前技术链路如下：

```text
数据源
→ 接入任务
→ Raw / Clean / Serving
→ 数据目录、质量和权限
→ 轻量语义模型
→ 业务关系图谱
→ Agent 受控工具调用
→ 来源、质量、不确定性和审计
```

只要这条链路稳定、清晰、可审计，平台就能为中小型制造企业提供简单高效可用的数据底座和 Agent 数据服务。
